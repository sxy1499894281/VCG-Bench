Problem

Pixel editing produces plausible diagrams, but professional diagram workflows need structure.

VCG-Bench shifts the unit of generation from raster pixels to executable mxGraph code, where nodes, edges, text, styles, and geometry can be inspected and edited.

Diagram-as-Pixels vs Diagram-as-Code Paper overview figure

Diagram-as-Pixels

Raster editing is stochastic and unstructured. It can blur lines, deform text, shift scale, break connectors, and hallucinate visual elements.

Diagram-as-Code

Symbolic XML keeps each diagram executable and editable. Local changes target explicit graph elements instead of re-sampling the whole image.

Benchmark Target

The benchmark separates visual structure recovery from code-level editing, making each capability measurable instead of blending them into one image score.

Framework

A unified benchmark for generation and editing around editable Draw.io diagrams.

The framework evaluates whether models can create valid mxGraph from diagrams, then use that structure for deterministic instruction edits.

VCG-Bench framework Required project figure
Task 1

Vision-to-XML Generation

Given a raster diagram image and structured caption, generate valid mxGraph XML that can render into a faithful diagram.

Task 2

Instruction-based XML Editing

Given existing XML, rendered context, and a natural-language instruction, predict localized XML patches for controlled edits.

Synthetic Data Pipeline

A machine-generation and human-verification loop for cold-starting open-source models.

VCG-Bench constructs Diagram-as-Code data by collecting candidate diagrams, synthesizing descriptions and XML, rendering candidates, and filtering for valid, visually aligned examples.

Task 1 and Task 2 data generation Synthetic data framework
Candidate Collection Start from diverse diagrams spanning academic, software, business, management, UIUX, and general domains.
XML Synthesis Generate structured captions and candidate mxGraph XML, then render the code back into diagrams for checks.
Verification Use parser validation, rendering verification, visual similarity, and human review to keep high-quality samples.

Benchmark Findings

Vision-to-code is the bottleneck; code-to-code editing becomes much more stable once XML exists.

The paper reports execution, visual fidelity, and semantic compliance through ESR, SCS, CodeXQA, SigLIP2, and XDRFR.

0.9626

Best Task 1 ESR

Closed-source systems lead Vision-to-XML generation, while open-source models often fail on executability even when they identify parts of the diagram.

0.0000

Open-source ESR failure mode

Some open models collapse on valid Draw.io generation, showing that visual recognition alone is not enough for executable structured output.

0.9405

Best Task 2 XDRFR

When the diagram is already available as XML, localized instruction editing is far more controllable and easier to verify.

ESRExecution success for parseable, renderable XML.
