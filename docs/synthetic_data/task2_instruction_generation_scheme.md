# 任务2指令生成方案（基于原子操作）

## 一、方案概述

### 1.1 核心策略

1. **使用1个统一Prompt**：一个Prompt同时生成Easy、Medium、Hard三个难度等级的指令
2. **基于原子操作的难度划分**：预先定义原子操作列表，通过原子操作数量控制难度
3. **对每张图片生成3条指令**：一次调用生成3条不同难度的指令

### 1.2 工作流程

```
对于每个样本（Gemini XML + Gemini 渲染图）：
└── 使用统一Prompt → 生成3条指令（Easy、Medium、Hard）
    ├── Easy指令：包含1-2个原子操作
    ├── Medium指令：包含3-4个原子操作
    └── Hard指令：包含5-7个原子操作

最终得到：每个样本有3条指令，分别对应Easy/Medium/Hard三个难度等级
```

## 二、原子操作定义与难度分级

### 2.1 原子操作列表

我们预先定义以下**14类原子操作**，每个操作都是图表编辑中最基本、不可再分的操作单元：

#### 类别1：节点属性修改（4个）
1. **修改节点颜色**：改变节点的填充颜色或边框颜色
   - 示例："把最上方的节点改成红色"

2. **修改节点形状**：改变节点的形状类型（矩形、圆形、菱形等）
   - 示例："把开始节点改成圆形"

3. **修改节点大小**：改变节点的宽度或高度
   - 示例："把标题框放大一些"

4. **修改节点文本**：改变节点内部显示的文本内容
   - 示例："把第一个节点的文字改为'开始'"

#### 类别2：节点结构操作（3个）
5. **删除节点**：删除一个节点（不处理连接线）
   - 示例："删除最左边的矩形"

6. **添加节点**：在指定位置添加一个新节点
   - 示例："在流程图开始位置添加一个新的节点"

7. **移动节点**：改变节点的位置
   - 示例："把标题往上移一点"

#### 类别3：连接线属性修改（3个）
8. **修改连接线颜色**：改变连接线的颜色
   - 示例："把所有箭头改成蓝色"

9. **修改连接线样式**：改变连接线的样式（实线、虚线、粗细等）
   - 示例："把连接线改成虚线"

10. **修改连接线箭头**：改变连接线的箭头样式
    - 示例："把单向箭头改成双向箭头"

#### 类别4：连接线结构操作（4个）
11. **删除连接线**：删除一条连接线
    - 示例："删除两个节点之间的连接线"

12. **添加连接线**：在两个节点之间添加一条连接线
    - 示例："在开始节点和结束节点之间添加一个箭头"

13. **重定向连接线**：改变连接线的起点或终点
    - 示例："把指向A的箭头改成指向B"

14. **更新连接线路径**：当节点移动后，更新连接线的路径坐标
    - 示例："调整连接线，让它们不重叠"

### 2.2 基于原子操作的难度划分

难度等级完全由**原子操作数量**决定：

- **Easy (简单)**：包含 **1-2个** 原子操作
  - 示例1："把最上方的节点改成红色"（1个操作：修改节点颜色）
  - 示例2："删除最左边的矩形，并删除所有与它相连的箭头"（2个操作：删除节点 + 删除连接线）

- **Medium (中等)**：包含 **3-4个** 原子操作
  - 示例1："把标题往右移一点，改成蓝色，并调整连接线"（3个操作：移动节点 + 修改节点颜色 + 更新连接线路径）
  - 示例2："删除开始节点，删除所有与它相连的箭头，把原本指向它的箭头改成指向第二个节点，并调整连接线"（4个操作：删除节点 + 删除连接线 + 重定向连接线 + 更新连接线路径）

- **Hard (困难)**：包含 **5-7个** 原子操作
  - 示例1："在流程图开始位置添加一个新节点，把它改成圆形，颜色改为红色，大小调大一些，在新节点和下一个节点之间添加箭头，设置箭头为虚线，并调整连接线布局"（7个操作：添加节点 + 修改节点形状 + 修改节点颜色 + 修改节点大小 + 添加连接线 + 修改连接线样式 + 更新连接线路径）
  - 示例2："删除中间的节点，删除所有与它相连的箭头，把指向它的箭头改成指向左边的节点，把从它出发的箭头改成从右边的节点出发，并调整所有连接线"（5个操作：删除节点 + 删除连接线 + 重定向连接线(输入) + 重定向连接线(输出) + 更新连接线路径）

### 2.3 原子操作组合原则

为了生成合理的指令，原子操作的组合应遵循以下原则：

1. **操作顺序逻辑性**：操作之间应有合理的先后顺序
   - ✅ 正确："把节点往右移，然后调整连接线"（先移动再更新）
   - ❌ 错误："调整连接线，然后把节点往右移"（顺序不合理）

2. **操作目标关联性**：多个操作应针对相关的元素
   - ✅ 正确："删除开始节点，并删除与它相连的箭头"
   - ❌ 避免："把最上方的节点改成红色，把最下方的节点放大"（两个不相关的节点）

3. **操作类型多样性**：尽量组合不同类型的原子操作，增加指令多样性
   - ✅ 鼓励：属性修改 + 结构操作（如"改颜色 + 删除节点"）
   - ❌ 避免：同一类型操作重复（如"把第一个节点改成红色 + 把第二个节点改成蓝色 + 把第三个节点改成绿色"）

### 2.4 指令的自然语言要求（重要）

**核心原则**：指令必须基于用户看到的**渲染图片**来描述，而不是基于XML代码。

#### ✅ 推荐的描述方式（按优先级排序）：

**最优先推荐**：直接使用节点上显示的文字内容

1. **节点文字内容描述（最自然、最准确）**：
   - "把'开始'节点改成红色"
   - "删除'数据处理'这个节点"
   - "把'用户登录'改成圆形"
   - "在'输入'和'输出'之间添加一个箭头"
   - "把'决策点'往右移一点"

   **为什么最优先**：用户看到图片时，最直观的就是节点上的文字，这是最自然的指代方式

2. **语义描述**（当节点虽然没有明确文字，但语义明确时）：
   - "开始节点"、"结束节点"
   - "标题文字"、"说明文字"

3. **位置描述**（当节点没有文字或文字不明确时使用）：
   - "最上方的节点"、"最左边的矩形"、"中间的圆形"
   - "左上角的文字"、"右下角的箭头"
   - "第一个节点"、"最后一个节点"

4. **视觉特征描述**（作为补充定位）：
   - "最大的矩形"、"最小的圆形"
   - "红色的节点"、"蓝色的箭头"
   - "虚线箭头"、"粗的连接线"

5. **相对位置描述**：
   - "在'开始'和'结束'之间的箭头"
   - "连接'输入'和'处理'的线"
   - "'标题'下面的节点"

#### ❌ 禁止的描述方式：

1. **技术性标识**：
   - ❌ "id为'node_1'的节点"
   - ❌ "edge_5"
   - ❌ "value属性为..."

2. **坐标描述**：
   - ❌ "坐标(200, 300)的节点"
   - ❌ "x=100, y=200位置"
   - ❌ "移动到(400, 500)"

3. **XML属性引用**：
   - ❌ "shape为rectangle的节点"
   - ❌ "style属性包含..."

#### 示例对比：

| ❌ 不自然（禁止） | ✅ 自然（推荐） |
|-----------------|---------------|
| 将id为'node_1'的节点颜色改为红色 | 把'开始'节点改成红色（最优先：使用节点文字） |
| 删除坐标(200, 300)的节点 | 删除'数据处理'这个节点（最优先：使用节点文字） |
| 将edge_5的颜色改为蓝色 | 把连接'输入'和'输出'的箭头改成蓝色（使用文字+相对位置） |
| 在(400, 500)位置添加节点 | 在'开始'和'结束'之间添加一个新节点（使用文字+相对位置） |
| 将node_2移动到node_3右侧 | 把'标题'框往右移到'处理'节点旁边（使用文字+相对位置） |

**注意**：如果节点有文字标签，优先使用文字来指代；如果节点没有文字，则使用位置、视觉特征等其他方式。

    if len(validated_instructions) < 3:

def batch_generate_instructions(samples):
    """
    批量为所有样本生成指令

    Args:
        samples: 样本列表

    Returns:
        包含所有指令的数据集
    """
    all_instructions = []

    for sample in samples:
        try:
            # 为每个样本生成3条指令
            instructions = generate_instructions_for_sample(sample)

            # 保存指令到样本对象
            sample.instructions = instructions
            all_instructions.extend(instructions)

            # 保存到文件（可选）

### 4.2 难度验证函数

```python
def validate_instruction_difficulty(instruction: dict) -> bool:
    """
    验证指令是否符合目标难度等级

    Args:
        instruction: 包含指令信息的字典

    Returns:
        是否通过验证
    """
    difficulty = instruction["difficulty"]
    operation_count = instruction["atomic_operation_count"]

    # 验证原子操作数量是否符合难度要求
    if difficulty == "Easy":
        if operation_count < 1 or operation_count > 2:
            log_warning(f"Easy指令的原子操作数量应为1-2，实际为{operation_count}")
            return False

    elif difficulty == "Medium":
        if operation_count < 3 or operation_count > 4:
            log_warning(f"Medium指令的原子操作数量应为3-4，实际为{operation_count}")
            return False

    elif difficulty == "Hard":
        if operation_count < 5 or operation_count > 7:
            log_warning(f"Hard指令的原子操作数量应为5-7，实际为{operation_count}")
            return False

    else:
        log_error(f"未知的难度等级: {difficulty}")
        return False

    # 验证原子操作ID是否在1-14范围内
    for op in instruction["atomic_operations"]:
        if op["operation_id"] < 1 or op["operation_id"] > 14:
            log_warning(f"无效的原子操作ID: {op['operation_id']}")
            return False

    # 验证是否包含指令文本
    if not instruction.get("instruction"):
        log_warning("指令文本为空")
        return False

    return True


def validate_operation_logic(instruction: dict) -> bool:
    """
    验证指令中原子操作的逻辑合理性（可选）

    检查：
    1. 操作顺序是否合理
    2. 操作目标是否关联
    3. 操作类型是否多样
    """
    operations = instruction["atomic_operations"]

    # 示例：检查"更新连接线路径"是否在"移动节点"之后
    for i, op in enumerate(operations):
        if op["operation_id"] == 14:  # 更新连接线路径
            # 检查前面是否有移动节点、删除节点等操作
            prev_ops = [o["operation_id"] for o in operations[:i]]
            if 7 not in prev_ops and 5 not in prev_ops:  # 7=移动节点, 5=删除节点
                log_warning("更新连接线路径应该在移动/删除节点之后")
                return False

    return True
```

## 五、与审稿人的说明

### 5.1 难度划分标准的可解释性

我们采用**基于原子操作数量的难度划分体系**，确保难度划分的简洁性和可解释性：

#### 原子操作定义
- 预先定义了14类原子操作，涵盖图表编辑中的所有基本操作
- 每个原子操作都是不可再分的最小操作单元
- 分为4大类别：节点属性修改、节点结构操作、连接线属性修改、连接线结构操作

#### 难度划分规则
- **Easy难度**：包含1-2个原子操作
- **Medium难度**：包含3-4个原子操作
- **Hard难度**：包含5-7个原子操作

这种划分方式：
- **简单直观**：难度完全由操作数量决定，容易理解和验证
- **可量化**：可以精确统计每条指令的原子操作数量
- **可复现**：审稿人可以轻松验证任何指令的难度等级
- **符合直觉**：操作数量越多，指令确实越复杂

### 5.2 自然语言指令的必要性

**核心设计理念**：本方案模拟真实用户场景。

#### 真实用户场景分析

用户的使用流程：
1. 用户在网上看到精美的流程图/PPT设计/论文插图
2. 用户让大模型生成XML代码来复现这个图片
3. 用户得到一张**渲染后的图片**（而不是XML代码）
4. 用户看着这张图片，用自然语言对话让大模型修改

用户看到的是**图片**，不是XML代码，所以：
- 用户不知道节点id是什么
- 用户不知道节点坐标在哪里
- 用户只能看到视觉特征：**节点上的文字**（最直观）、位置、颜色、形状、大小

#### 为什么必须使用自然语言

| 场景 | 用户会说 | 用户不会说 |
|------|---------|-----------|
| 修改节点颜色 | "把'开始'节点改成红色"（最自然：使用节点文字） | "将id为'node_1'的节点颜色改为红色" |
| 删除节点 | "删除'数据处理'这个节点"（最自然：使用节点文字） | "删除坐标(200, 300)的节点" |
| 修改连接线 | "把连接'输入'和'输出'的箭头改成蓝色"（使用文字） | "将edge_5的颜色改为蓝色" |
| 添加节点 | "在'开始'和'结束'之间添加一个新节点"（使用文字） | "在(400, 500)位置添加节点" |

**核心原则**：优先使用节点上的文字内容来指代节点，这是最符合用户直觉的方式。

如果评测使用技术性描述（id、坐标），则：
- ❌ 不符合真实用户场景
- ❌ 无法评测模型的视觉理解能力
- ❌ 无法评测模型识别和理解节点文字的能力
- ❌ 无法评测模型的自然语言理解能力

如果评测使用自然语言描述（优先使用节点文字），则：
- ✅ 符合真实用户使用场景
- ✅ 评测模型的OCR和文字识别能力（能否识别节点上的文字）
- ✅ 评测模型的视觉理解能力（能否根据"最上方的"定位节点）
- ✅ 评测模型的自然语言理解能力（能否理解"开始节点"的语义）
- ✅ 评测模型的多模态理解能力（文字+位置+视觉特征综合理解）
- ✅ 更有实际应用价值

### 5.3 统一Prompt的优势

相比使用3个专用Prompt，我们采用1个统一Prompt的优势包括：

1. **一致性更强**：
   - 所有指令都由同一个Prompt生成，确保风格一致
   - 避免不同Prompt之间的语言风格差异

2. **效率更高**：
   - 一次LLM调用生成3条指令，节省API调用次数
   - 减少总体生成时间（从3次调用降为1次）

3. **上下文相关**：
   - LLM可以在生成3条指令时相互参考，避免重复操作相同的元素
   - 确保3条指令具有多样性

4. **维护成本低**：
   - 只需维护1个Prompt，而不是3个
   - 修改难度标准时，只需更新1处

### 5.4 数据集的平衡性与多样性

通过统一Prompt一次性生成3条不同难度的指令，确保：

1. **难度覆盖**：
   - 每个样本都有Easy/Medium/Hard三个难度等级的指令
   - 三个难度等级的指令数量严格相等（各占33.3%）

2. **原子操作多样性**：
   - 14类原子操作可以灵活组合
   - 统计分析可以验证所有原子操作类型都被充分使用

3. **指令多样性**：
   - 同一图表的3条指令涉及不同的元素或操作类型
   - 避免3条指令过于相似

4. **描述方式多样性**：
   - 位置描述、视觉特征、语义描述等多种方式
   - 评测模型在不同描述方式下的理解能力

### 5.5 可执行性保证

我们的方案设计了多层验证机制：

1. **原子操作验证**：
   - 验证operation_id是否在1-14范围内
   - 验证原子操作数量是否符合难度要求

2. **逻辑顺序验证**：
   - 检查操作顺序是否合理（如"更新连接线路径"应在"移动节点"之后）
   - 检查操作目标是否关联

3. **元素存在性验证**：
   - 虽然使用自然语言描述，但需要验证描述的元素在渲染图中是否可以唯一确定
   - 例如"最上方的节点"在图表中必须有明确对应
   - 确保指令可以在实际图表上执行
